01 · Introduction

Professional Self-Assessment

When I started the Computer Science program at Southern New Hampshire University, I was already running a business. I own a luxury event design and planning company, and for most of this degree I was writing code in the mornings and planning celebrations in the evenings. That balance shaped how I think about software. Every assignment in this program sat next to a real deadline, a real client, and a real budget, and I learned early that good engineering is mostly about making smart decisions under constraints.

Completing this coursework and building this ePortfolio helped me define what I actually want from my career. I am positioning myself for hybrid roles that combine full stack development with technical project management, ideally at a product company where I can own delivery from requirements through release. The capstone asked me to look back at my own work with a critical eye, decide what was worth improving, plan those improvements, and then ship them on a schedule. That cycle of review, plan, build, and verify is exactly the work of a technical project manager, and it is the strongest evidence I can offer that I am ready for that work.

Collaborating in a team environment

Collaboration in software is structured, and I have come to appreciate that structure. The code review I produced for this capstone follows the same format I would use on a real team: walk through existing functionality, analyze the code honestly, and present a plan that stakeholders can react to before any code changes. Courses like CS 250 taught me how agile ceremonies, sprint planning, and retrospectives turn a group of individuals into a delivery team. My business gives me daily practice as well. Coordinating vendors, venues, and clients toward a single event date is sprint planning by another name, and it taught me to communicate status early and surface risks before they become emergencies.

Communicating with stakeholders

I write and present for different audiences constantly, and I treat that as a core technical skill rather than a soft one. In CS 370 I wrote a GDPR white paper aimed at an executive audience, which meant translating regulatory and technical detail into business consequences. My capstone code review is aimed at peers and a manager, so it explains not just what the code does but why each enhancement is worth the time. With my own clients, I translate logistics and contracts into plain language every week. The common thread is the same: meet the audience where they are, and never hide a decision behind jargon.

Data structures and algorithms

CS 300 was the course that made algorithmic trade-offs concrete for me. I built an advising program for ABCU around a binary search tree, and in the capstone I returned to it with sharper eyes. Loading sorted course data into a plain BST degrades it into a linked list, so I rebuilt construction median-first to keep the tree balanced and searches logarithmic. I added depth-first search to detect circular prerequisites, which the original code never considered, and I replaced linear prerequisite scans with an unordered set for constant time validation. Each change was a deliberate trade-off between memory, speed, and clarity, and being able to reason about those trade-offs out loud is what separates writing code from designing solutions.

Software engineering and databases

CS 465 had me build Travlr Getaways, a full MEAN stack travel application with an Express API, a MongoDB data layer, and an Angular single page admin interface. Two of my three capstone enhancements live in this artifact. On the engineering side I consolidated fragmented routing, moved the API to async and await through a shared handler, and centralized error handling so failures are consistent and observable. On the database side I corrected schema types so numeric fields are actually numeric, added validation middleware so bad data is rejected at the boundary, built aggregation pipeline endpoints that compute statistics inside MongoDB rather than in application code, and added a compound index with timestamps to support the new query patterns. Working across one artifact at both the engineering and data layers showed me how design decisions ripple through an entire stack.

Security

The program built a security mindset in me that I now apply by default. In the Travlr enhancement I removed a hardcoded JWT secret and made the application refuse to start without one supplied through the environment, because a missing secret should be a loud failure rather than a quiet vulnerability. I added a mass-assignment whitelist so clients can only set the fields I intend them to set, and server side validation so the database never has to trust the client. The habit underneath all of it is simple: assume input is hostile, assume secrets leak, and design so that mistakes fail safely.

How the artifacts fit together

This portfolio contains two artifacts carrying three enhancements. Travlr Getaways demonstrates breadth: a single full stack application improved at the software engineering layer and again at the database layer, with security woven through both. The ABCU advising program demonstrates depth: focused algorithmic reasoning in C++ where every line answers to performance and correctness. Together they show that I can design a system, build it, secure it, measure it, and explain it to the people who depend on it. The sections that follow present the code review that planned this work and the narratives that reflect on what each enhancement taught me.

Enhancement One · Software Design & Engineering

Travlr Getaways: hardening a full stack travel platform

Artifact: CS 465 Full Stack Development · MEAN stack · Node, Express, MongoDB, Angular

Travlr Getaways is a travel booking application I built in CS 465, with a customer facing site, an Express REST API, and an Angular admin SPA. The original worked, but it carried the habits of a course project: duplicated route logic, callback style handlers, inconsistent errors, and a JWT secret sitting in source code.

The enhancement refactored the API into a consistent, secure, production shaped service. The narrative reflects on why centralizing failure paths and forcing secrets into the environment changes the character of an application, not just its style.

Enhancement Two · Algorithms & Data Structures

ABCU Advising: balancing the tree, catching the cycles

Artifact: CS 300 Data Structures & Algorithms · C++ · Binary Search Tree

The ABCU Advising Program loads a course catalog from CSV into a binary search tree and lets advisors search courses and print prerequisites in order. The original implementation was correct for happy paths but fragile everywhere else: sorted input degraded the tree to O(n) searches, circular prerequisites would never be detected, and prerequisite validation scanned linearly.

The enhancement turned a passing assignment into a defensible piece of algorithm design, where every structure is chosen for a measurable reason and every input is treated as untrusted.

Enhancement Three · Databases

Travlr Getaways: pushing the work into MongoDB

Artifact: CS 465 Full Stack Development · MongoDB · Mongoose, aggregation pipelines

The third enhancement returned to Travlr Getaways at the data layer. The original schema stored prices and durations as strings, accepted any payload the client sent, and answered every analytical question by pulling documents into application code.

The enhancement corrected the schema so the data means what it says, added validation middleware so the database boundary defends itself, and introduced aggregation pipeline endpoints that compute trip statistics and upcoming departures inside MongoDB, supported by a compound index and timestamps. The narrative reflects on treating the database as an active participant in the application rather than a passive store.
